# Template: Browser Extension

Use this template when building a Chrome/Firefox/Edge extension.

## Stack

| Layer | Technology | Why |
|-------|-----------|-----|
| Framework | WXT (Vite-based) | File-based entrypoints, HMR, cross-browser, framework-agnostic |
| Language | TypeScript | Type safety |
| UI | React + Tailwind CSS | For popup, options, sidepanel |
| Manifest | V3 (auto-generated by WXT) | Required for Chrome since 2024 |
| Storage | chrome.storage.local/sync | Synced across devices |
| Messaging | WXT typed messaging | Between content script в†” background |
| Build | Vite (via WXT) | Fast builds, HMR for development |
| Testing | Vitest | Unit + integration tests |
| Deploy | Chrome Web Store + Firefox Add-ons | Manual upload or CI |

## Alternative: Plasmo

| When | Use |
|------|-----|
| React-first team, simpler project | Plasmo |
| Multi-framework, complex extension | WXT |
| Minimal deps, full control | Vite + CRXJS plugin |

## Key Patterns

- **Manifest V3** вЂ” service workers instead of background pages (no persistent state)
- **Content Script isolation** вЂ” runs in page context, communicate via messages
- **Typed messaging** вЂ” define message types, WXT handles serialization
- **Storage abstraction** вЂ” wrap `chrome.storage` with typed getters/setters
- **Permissions** вЂ” request minimum permissions, use optional permissions
- **CSP compliance** вЂ” no inline scripts, no `eval()`, no remote code

## Manifest V3 gotchas:

- Service worker can be killed after 5 minutes вЂ” don't rely on in-memory state
- Use `chrome.alarms` instead of `setInterval` for background tasks
- `fetch` works in service worker, `XMLHttpRequest` does not
- Dynamic `import()` not supported in content scripts

## Agents should know:

- PM: Extension = small, focused utility. One clear value prop.
- Designer: Popup is 400x600 max. Minimal UI. Fast interactions.
- Engineer: Service worker lifecycle, message passing, storage API, CSP rules
- QA: Test on Chrome + Firefox, test service worker restart, test permissions
